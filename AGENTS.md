# 英语小程序底层规则

本文件是 `D:\芸\英语` 的唯一主规则。这个文件夹只服务英语精读微信小程序。每次新会话、每次开始思考任务前，先读本文件，再行动。

## 最高约束

1. 用户当前明确指令优先。
2. 不改用户没要求改的程序代码、架构、依赖、配置和生成产物。
3. 先做检查和判断，再改文件；低风险、可验证的小改动可以直接执行，涉及架构、付费、删除、API Key、后端方案或视觉大方向时先问用户。
4. 重要决策写进文档，不只留在聊天记录里。

## 每次开工前

- 先读本文件。
- 判断任务类型：问答、排查 bug、UI/交互、架构/后端、文档、构建验收。
- 再按当前平台的 skill 规则判断是否需要 skill；不要扫描或读取全部 skill。
- 只读取用户明确要求、任务直接相关、或平台规则强制要求的 skill。
- 使用了哪个 skill、为什么用，要简短反馈给用户；不相关的插件和 skill 不使用。
- 如果只是本地文件检查、普通编辑或简单问答，不要为了“看起来完整”调用外部插件。

## Skill 与插件

- skill 用来决定怎么做事；插件用来调用外部能力。
- 插件缓存里能看到的内容，不等于当前会话一定能调用；当前可调用工具以实际暴露的工具和 `tool_search` 为准。
- 每次任务前精准判断是否需要插件：
  - Figma：设计稿、设计系统、界面还原。
  - GitHub：PR、issue、CI、代码评审、发布分支。
  - Browser/Chrome：页面预览、截图、交互验证。
  - Product Design：产品流程、UX 审计、界面方案。
  - Google Drive：Docs、Sheets、Slides、Drive 文件。
  - Creative Production / ImageGen：品牌视觉、图片、广告、素材。
  - Documents/PDF/Sheets/Slides：本地文档、表格、演示处理。
- 插件只在任务真的需要时使用，避免浪费 token 和引入无关流程。
- 用户提到某个插件或 skill 时，先判断它是否对当前目标有实际帮助；如果没有帮助，就不调用，并保持工作推进。
- 用了插件，要告诉用户：用了哪个、解决什么问题。

## 多角色审视

复杂功能、新功能、明显改进、架构调整前，至少快速从这些角度过一遍：

- 产品经理：这个改动是否服务英语精读主路径。
- AI vibe coding 负责人：是否小步快跑、少改动、可回退。
- UX 设计师：是否减少打扰，尤其不破坏阅读体验。
- 测试工程师：最小验收路径和风险点是什么。
- 架构师：是否遵守页面、service、repository 边界。
- 安全负责人：API Key、用户数据、云函数边界是否安全。
- GitHub PR reviewer：改动是否可审，是否混入无关重构。
- 成本管理员：是否节省 token、AI 调用和重复读取。

## Token 节省

- 先读目录、规则、关键文件，再读大文件。
- 不全文读取生成产物，除非验收 WXML/构建输出必须看。
- 长文档只读与当前任务相关部分。
- 能用 `rg` 定位就不要盲目打开整堆文件。
- 不把 `using-superpowers/SKILL.md` 全文复制进本文件，只保留引用和执行要求。
- 长会话中，把稳定结论写入本文件或 `docs/`，减少重复解释。
- 同一轮只读当前任务需要的规则和源码，不重复读取已确认无关的大文件。
- 结论能沉淀成规则时，走“规则候选评估”机制，不在聊天里反复解释。

## 规则候选评估

对话中如果出现新的好规则、踩坑经验或稳定工作方式，先评估，不自动写入文档。

满足以下多数条件时，向用户询问是否加入规则：

- 以后很可能重复用到，而不是一次性偏好。
- 能减少返工、误操作、token 消耗或验收混乱。
- 与英语精读小程序长期方向一致。
- 能写成简短、可执行、可验证的规则。
- 不和用户当前明确要求、平台规则、现有架构边界冲突。

询问格式保持简短：

“这条规则值得沉淀吗？建议加入 `<文件路径>`：`<规则一句话>`。”

规则沉淀位置：

- 全局高频规则放 `D:\芸\英语\AGENTS.md`。
- 微信验收、DevTools、阅读页专项规则放 `class-english-miniprogram/docs/wechat-miniprogram-working-rules.md`。
- 架构、目录、后端边界放 `class-english-miniprogram/docs/frontend-architecture-map.md` 或产品 PRD。
- 临时结论、一次性偏好、未验证想法不写入规则。

如果一轮对话出现多条候选规则，优先在阶段结束时合并询问，不频繁打断用户。

## 项目定位

- 本项目优先交付微信小程序。
- 微信开发者工具和真机体验是最终验收面。
- H5 预览只做辅助，不作为阅读页交互验收标准。
- 产品核心是英语精读：读文章、懂句子、查单词、收藏生词、复习巩固。
- 体验目标是精致、克制、干净、阅读优先。

## 工作入口

- 微信开发者工具导入目录：`D:\芸\英语\class-english-miniprogram`。
- 主要源码在 `class-english-miniprogram/src/`。
- 优先修改 `src`，再构建到 `dist/build/mp-weixin`，构建脚本会同步到根目录兜底小程序产物。
- 不直接手改 `dist/build/mp-weixin`。
- 不直接手改根目录同步出来的 `pages`、`utils`、`common`、`static` 产物。
- 根目录同步产物用于微信开发者工具直接编译显示，不是主要编辑入口。
- 用户新构建后在微信开发者工具里点“编译”即可，不需要重新导入。

## 架构边界

- 页面只负责 UI 状态和交互编排。
- 业务入口放在 `src/services/`。
- 数据访问放在 `src/repositories/`。
- 本地 mock、缓存、云数据库、云函数的切换优先在 repository 层处理。
- 当前微信端如果需要稳定编译路径，可以保留 `src/utils/*Service.ts` 作为小程序安全 facade；不要重新引入会导致编译后模块找不到的不稳定引用。
- 不把文章数据、翻译缓存、正文解析算法继续塞回页面文件。
- API Key 只能放云函数环境变量，不能进入 `src` 前端代码。

## 阅读页规则

- 阅读页交互以微信小程序表现为准。
- 句子翻译用紧凑气泡/弹窗，不占正文布局空间，不遮挡当前句子。
- 单词查询保持小学习卡片感，不做成大底部抽屉。
- 弹窗必须轻量、非阻塞，不能盖住正在学习的句子。
- 默认阅读字号保持克制，约 `16px` 字号、`28px` 行高。
- 旧底部三按钮工具条 `reader-toolbar` 不允许回归。
- 文章图片和封面使用本地资源，主题色要与文章图片或文章主题呼应。

## 新功能或改进前

- 先检查已有文档：`docs/product-requirements.md`、`docs/frontend-architecture-map.md`、`docs/wechat-miniprogram-working-rules.md`。
- 先检查相似页面、相似 service、相似 repository。
- 先确认不会改变用户没要求改的架构。
- 先说明计划改哪些文件。
- 小修复直接做；中等体验改动先给 2-3 行方案；大改版、架构、后端、数据库、付费和 API Key 接入必须先确认。

## 反复失败处理

- 同一个问题连续两次修改后仍无效，停止继续猜测和叠加修改。
- 先定位证据链：源码是否改到、构建产物是否更新、根目录兜底产物是否同步、微信开发者工具是否加载同一份文件、截图是否显示临时版本标记。
- DevTools 显示旧 UI 时，优先检查项目路径、`project.config.json`、根目录产物、`dist/build/mp-weixin`、WXML 中的关键标记，而不是继续改样式。
- 修复失败时，先解释当前证据和下一步验证点，再继续动代码。

## 构建与验收

- 修改阅读页 UI 或交互后，运行 `npm.cmd run build:mp-weixin`。
- 阅读页改动后运行 `npm.cmd run check:ui`。
- TypeScript 相关改动需要运行 `npm.cmd exec vue-tsc -- --noEmit`。
- 完成前检查 `dist/build/mp-weixin` 的 WXML 事件结构，确认交互已正确编译。
- 微信开发者工具里新构建后直接点“编译”即可，不需要重新导入。
- 关键验收标记：不出现 `reader-toolbar`；出现预期的新 UI 标记；句子翻译不挤占正文；单词卡片不是大抽屉；正文默认约 `16px/28px`。
- 最终回复必须告诉用户：
  - 改了哪些文件。
  - 用了哪些 skill/插件。
  - 跑了哪些检查，哪些没跑。
  - 用户该如何验收。
