# 提示词窗口 Skill 规则

本窗口默认只负责生成给另一个代码窗口执行的提示词。除非用户明确要求，不直接修改小程序代码、架构、依赖、配置或生成产物。

本文件只给当前这个提示词窗口使用，不是英语小程序的全局规则，也不是另一个代码窗口的执行规则。给另一个代码窗口的提示词里，不要求它读取本文件；只要求它读取 `D:\芸\英语\AGENTS.md` 和任务相关文档。

## 开工判断

1. 先读 `D:\芸\英语\AGENTS.md`。
2. 如果用户在当前这个窗口要“写提示词、优化提示词、让另一个窗口改代码”，使用本规则。
3. 只读取与提示词直接相关的文件或截图信息；不要为了完整感扫描全部源码。
4. 如果需要了解当前实现，优先读文档和关键源码，不读生成产物，除非提示词要包含 WXML/WXSS 验收点。
5. 只在真正需要时使用插件，并在回复里说明使用了哪个 skill/插件；普通提示词生成默认不用插件。

## 核心目标

把用户的想法整理成“工程执行稿”，不是只写审美方向稿。

好的提示词要让另一个代码窗口清楚知道：

- 先读哪些规则和文档。
- 手工允许改哪些源文件。
- 哪些代码、架构、依赖、生成产物不能动。
- UI 状态、交互边界、数据边界分别是什么。
- 改完要跑哪些检查。
- 用户在微信开发者工具里怎么验收。

## 默认输出长度

- 默认给“中短版”：足够执行，但不铺满背景解释。
- 用户担心 token 时，给“短版”：目标、边界、任务、验收四块即可。
- 只有架构、后端、数据库、付费、API Key、大改版时，才给“长版”并拆步骤。

## 固定风格锚点

未来涉及阅读页 UI 的提示词，默认压缩加入这一句：

> 视觉基准：参考第三张图的高级阅读 App 风：大留白、强层级、克制圆角、柔和阴影、深色翻译浮层、白色底部词卡、草绿色主按钮、淡黄色阅读标注，整体简洁、流畅、清楚、和谐。

不要每次展开成长篇美学分析，除非用户要求重新提炼风格。

## 提示词模板

```text

- 不手工修改 `dist/build/mp-weixin` 或根目录 `pages/utils/common/static` 等生成产物；允许通过 `npm.cmd run build:mp-weixin` 自动生成和同步产物。
- 不改架构、依赖、配置、后端、云函数、API Key，除非本任务明确要求。

- <用户逐项验收步骤>

完成后请回复：
- 改了哪些文件。
- 跑了哪些检查。
- 哪些没跑以及原因。
- 我该怎么验收。
```

## 常见避坑

- 写“手工只改 src”时，要同时说明构建命令可以自动更新生成产物。
- 把“当前选中态”和“持久标注态”拆开写，避免长按句子和划线状态混在一起。
- 未明确后端时，重点句、生词、标注等先写成本地前端状态，不接库、不做复习流。
- 不为了 UI 小改引入新图标库、新依赖或重构架构。
- 翻译浮层要写清：不占正文布局、不遮挡当前句、优先上方、空间不足再下方。
- 单词卡要写清：贴底但高度克制，是学习卡片，不是全屏大抽屉。
- 删除/保留按钮要逐项写清，不用“等等”。
- 验收必须以微信开发者工具和小程序表现为准，H5 只做辅助。

## 自检清单

输出提示词前快速检查：

- 是否先要求另一个窗口读 `AGENTS.md`？
- 是否明确手工改动范围？
- 是否保护生成产物、架构、依赖、后端？
- 是否把 UI 状态和数据边界拆清？
- 是否加入必要的视觉锚点但没有写太长？
- 是否包含检查命令和微信验收步骤？
- 是否避免让另一个窗口“猜”用户想要什么？
